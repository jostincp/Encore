### Principio de Microservicios: DRY (Don't Repeat Yourself)

En una arquitectura de **microservicios**, cada servicio debe ser **independiente** PERO pueden compartir:
- **Tipos/Interfaces comunes** (para comunicación)
- **Modelos de dominio compartidos** (entidades centrales)
- **Utilidades** (validación, logging, etc.)

### El Problema Actual

```
❌ ANTES (incorrecto):
backend/
├── auth-service/
│   └── src/models/Bar.ts  ← Solo auth-service puede usar Bar
├── music-service/
│   └── src/controllers/queueController.ts  ← Intenta usar BarModel pero NO lo tiene
└── queue-service/
    └── (también necesitaría Bar)
```

**Resultado**: Cada servicio que necesite [Bar](cci:2://file:///c:/www/Encore/backend/shared/types/index.ts:31:0-41:1) tendría que:
1. Duplicar el código (malo ❌)
2. Hacer llamadas HTTP entre servicios (lento, complejo ❌)
3. No compilar (como está pasando ahora ❌)

### La Solución: `shared`

```
✅ DESPUÉS (correcto):
backend/
├── shared/
│   └── models/
│       └── Bar.ts  ← TODOS los servicios pueden importarlo
├── auth-service/
│   └── (importa desde ../shared/models/Bar)
├── music-service/
│   └── (importa desde ../../../shared/models/Bar)
└── queue-service/
    └── (importa desde ../../../shared/models/Bar)
```

### ¿Por qué [Bar](cci:2://file:///c:/www/Encore/backend/shared/types/index.ts:31:0-41:1) específicamente debe estar en `shared`?

[Bar](cci:2://file:///c:/www/Encore/backend/shared/types/index.ts:31:0-41:1) es una **entidad de dominio central** porque:

1. **`auth-service`** lo necesita para:

   - Validar que el bar existe antes de agregar canciones
   - Obtener configuración del bar (auto-aprobar canciones, etc.)

3. **`queue-service`** probablemente lo necesite para:

# 🎯 ¿Qué son los Path Aliases?

Los **path aliases** (alias de rutas) son **atajos/shortcuts** que defines en TypeScript para evitar escribir rutas relativas largas y confusas.

## ❌ Problema: Rutas Relativas Largas

Sin path aliases, tu código se ve así:

```typescript
// En backend/music-service/src/controllers/queueController.ts
import { BarModel } from '../../../shared/models/Bar';
import logger from '../../../shared/utils/logger';
import { validatePaginationParams } from '../../../shared/utils/validation';
import { AuthenticatedRequest } from '../../../shared/types/auth';
```

**Problemas:**
- Difícil de leer: `../../../` es confuso
- Frágil: Si mueves el archivo, todos los imports se rompen
- Propenso a errores: Fácil equivocarse contando `../`

## ✅ Solución: Path Aliases

Con path aliases, el mismo código se ve así:

```typescript
// En backend/music-service/src/controllers/queueController.ts
import { BarModel } from '@shared/models/Bar';
import logger from '@shared/utils/logger';
import { validatePaginationParams } from '@shared/utils/validation';
import { AuthenticatedRequest } from '@shared/types/auth';
```

**Beneficios:**
- ✅ Más limpio y legible
- ✅ No importa dónde esté el archivo
- ✅ Fácil de mantener
- ✅ Autocompletado funciona mejor en IDEs

---

## 🔧 Cómo Configurar Path Aliases

### 1. En `tsconfig.json`

Tu archivo `backend/music-service/tsconfig.json` ya tiene una configuración parcial (líneas 36-40):

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@shared/*": ["../shared/*"]
    }
  }
}
```

**Esto significa:**
- `@/*` → Se resuelve a `backend/music-service/src/*`
- `@shared/*` → Se resuelve a `backend/shared/*`

### 2. Para que funcione en Runtime con ts-node

Necesitas `tsconfig-paths` (líneas 54-56):

```json
{
  "ts-node": {
    "require": ["tsconfig-paths/register"]
  }
}
```

**Esto hace que:**
- TypeScript en **compilación** entienda los aliases
- `ts-node` en **runtime** también los entienda
